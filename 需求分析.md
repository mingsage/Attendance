# 班级考勤系统 — 需求分析

---

## 一、产品定位

### 1.1 服务场景

高校日常课堂考勤是教学管理的基础环节。当前主流方式分为三类：传统点名（耗时长、无法杜绝代答）、刷卡/扫码（需携带卡片或手机，仍可代刷）、生物识别（人脸/指纹，技术门槛高但防伪能力强）。本项目选择**人脸识别路线**，核心原因在于面部信息天然与个人绑定，无需额外携带设备，且可结合活体检测技术防止照片、视频等低成本伪造手段。

系统的核心使用场景有两个。**课堂签到**：教师在课前输入课程名称，开启摄像头考勤，学生依次或排队经过摄像头完成签到，整个过程自动记录、可追溯。**活动合照识别**：班级活动结束后，教师上传合照，系统自动识别照片中的学生并生成参与名单。

### 1.2 服务对象

系统面向两种角色，各自有不同的职责边界和使用诉求。

**教师**是系统的管理者和主要操作者。教师需要在课前启动考勤、活动后上传合照、随时查看考勤记录和统计数据、管理学生档案和人脸库、导出报表用于教学管理或提交。教师的核心诉求是效率与可靠性——签到流程不应成为课堂的负担，结果应可信、可查、可导出。

**学生**是系统的参与者。学生需要完成摄像头签到、查看自己的考勤记录、维护个人信息（人脸上传、密码修改）。学生的核心诉求是便捷与隐私——签到操作简单，只能看到自己的数据，个人信息可自助管理。

### 1.3 任务硬性要求（评分对照）

根据课程实验任务和评分标准（满分 100 分），系统需满足以下硬性指标：

| 评分维度 | 分值 | 核心要求 |
|---------|------|---------|
| 架构要求 | 10 | BS 分层清晰、浏览器兼容、API 规范、数据库合理 |
| 基础考勤 | 25 | 摄像头采集、手动/自动捕捉、活体检测抗欺骗、人脸库管理、数据导出 |
| 合照识别 | 9 | 批量人脸检测识别、准确率 ≥85%、活动参与记录、统计报表 |
| 情绪分析 | 8 | 同步面部特征提取、7 类情绪分类、完整数据记录、前端统计 |
| 系统安全 | 18 | 抗照片伪造、抗视频伪造、角色权限区分 |
| 实验报告 | 30 | 结构完整的设计报告、源码完整可部署 |

## 二、系统架构

### 2.1 整体架构

系统采用前后端分离的 BS 架构。前端运行在浏览器中，负责界面渲染、用户交互、摄像头调用和图像采集。后端运行在服务器，负责人脸检测与识别、活体检测、情绪分析、数据持久化和统计分析。前后端通过 RESTful API 基于 HTTP 协议通信，数据交换格式为 JSON（文件上传使用 multipart/form-data）。

分层设计的核心原则是**关注点分离**：前端不处理任何 AI 推理逻辑，后端不关心界面渲染。前端专注于流畅的用户体验和清晰的状态反馈；后端专注于算法管线的高效执行和数据的一致性保障。各层职责明确，边界清晰。

前端内部进一步分层：视图层（页面组件）负责界面布局和交互，状态层（Pinia store）管理登录状态和用户信息，路由层（Vue Router）控制页面跳转和权限守卫，API 层（Axios 封装）统一管理后端接口调用和请求拦截。后端内部分层：路由层（API endpoints）接收请求和参数校验，服务层（services）封装人脸、活体、情绪等 AI 能力，模型层（models）定义数据库结构，核心层（core）管理配置、数据库连接和安全组件。

### 2.2 数据库设计

数据库围绕"学生"展开，采用关系型模型。核心实体包括学生（基本档案 + 人脸特征）、用户（登录账号）、考勤记录、情绪记录、活动参与记录。表之间通过外键关联：用户关联到学生（一个学生最多一个账号），考勤记录关联到学生（一条记录属于一个学生），情绪记录关联到学生，活动参与记录关联到学生。索引覆盖常见查询字段（学号、课程名、时间戳、状态），确保查询性能。

人脸特征以二进制形式直接存入学生记录。在班级规模（数十到数百人）下，单次全库遍历比对的开销完全可接受，无需引入专用向量数据库。特征的存储格式为归一化浮点向量，入库前和出库时有统一的编解码流程。

### 2.3 API 设计

API 遵循 RESTful 风格，接口路径以资源名称为核心（如 `/students`、`/attendance/records`）。GET 用于查询，POST 用于创建和执行操作，PUT 用于更新，DELETE 用于删除。所有需要身份验证的接口通过 JWT 令牌保护：请求头携带 Bearer 令牌，后端解析令牌获取当前用户身份和角色。

接口输入输出通过类型化 Schema 定义和数据校验。文件上传接口额外进行格式白名单校验和大小限制。错误响应统一包含 HTTP 状态码和可读的错误描述。对于查询接口，支持通过查询参数传递筛选条件（关键词、状态、课程名等）和分页参数。

## 三、用户权限区分

系统定义教师和学生两种角色。权限设计遵循最小权限原则——每种角色只能访问其职责范围内的功能和数据。

**教师**拥有全部管理权限。可进行考勤签到、查看全部学生的考勤记录（支持按课程、日期、关键词、状态筛选）、管理系统学生档案（增删改查）、单张或批量录入人脸照片并自动创建登录账号、上传合照进行批量识别、查看情绪统计和活动统计、导出考勤明细和统计报表。

**学生**权限限于个人操作范围。可进行考勤签到、查看考勤记录（可见全部记录以了解签到进度，但无法管理他人数据）、管理个人信息（查看学籍信息、上传或拍照录入人脸、修改登录密码）。学生无法访问学生管理、合照识别等教师专属功能。

权限控制的实现分为三层。**认证层**：用户登录时后端验证密码（bcrypt 哈希存储），签发 JWT 令牌（包含用户 ID 和角色）。令牌具有过期时效，每次请求通过拦截器自动附加。**接口层**：通过依赖注入进行角色校验，标记为教师专属的接口在学生令牌访问时返回权限拒绝。**路由层**：前端路由配置中声明每条路由允许的角色，导航守卫在页面跳转时检查角色是否匹配，不匹配则重定向。三层防护确保权限边界不会因绕过其中一层而失效。

学生用户通过批量导入人脸时自动创建——系统根据学号生成登录账号（默认密码），无需教师手动逐一创建。学生可使用自己的账号登录系统，进行签到和个人信息维护。

## 四、数据处理

### 4.1 图像采集与传输

前端通过浏览器标准 API 调用摄像头，获取高分辨率视频流。用户操作（手动按钮点击或自动检测触发）时从视频中截取单帧，压缩后通过 HTTP 上传至后端。压缩参数在画质和传输速度之间取平衡——过高的画质增加传输耗时，过低的画质影响后续检测和识别精度。

后端接收图片后首先进行格式校验：检查 MIME 类型和文件后缀是否在允许的范围内，不在则直接拒绝。系统接受常见的 JPEG、PNG 格式，同时兼容 WebP、AVIF、BMP、TIFF 等格式——对于非主流格式，后端自动转换为统一格式后再处理，而非直接拒绝。文件大小设有明确上限，超过限制的请求被拦截。

校验通过后，图片被解码为可供算法处理的数值矩阵。解码过程优先使用高效的图像处理库，失败时通过通用图像库进行格式转换后重新解码，确保尽可能多的合法图片能被正确处理。

### 4.2 人脸检测与质量评估

人脸检测是图像进入系统后的第一步处理。检测器在画面中定位所有人脸的位置和大小，返回矩形边界框。主检测采用 ONNX 深度学习模型（轻量级，适合 CPU 推理），在检测精度和速度之间取得良好平衡。当模型文件缺失或初始化失败时，系统自动降级为传统级联分类器作为保底方案，确保在任何部署环境下都能运行。

检测结果经过质量评估后决定是否继续后续流程。质量评估考察两个维度：人脸尺寸是否足够（过小的人脸缺乏足够的像素信息，识别精度会显著下降）、人脸位置是否完整（部分在画面外的人脸特征不完整，容易导致误匹配）。评估不通过的直接拒绝，并向用户返回具体的失败原因。

考勤场景与合照场景的检测策略不同。考勤场景要求画面中有且仅有一张主要人脸——检测到多张时取面积最大者，检测不到时直接拒绝，不做盲目猜测。合照场景则检测全部人脸，不限制数量，每张人脸都会进入后续的识别流程。

### 4.3 特征提取

特征提取将人脸图像转化为可供数学比对的数值向量——每个人脸被映射为高维空间中的一个点，点与点之间的距离反映人脸的相似程度。

提取过程经过多步预处理。首先利用人脸上的关键点进行几何对齐——通过仿射变换将任意姿态的人脸归一化到标准角度，消除头部偏转、倾斜等姿态差异的影响。然后进行光照归一化——通过自适应直方图均衡化补偿不均匀光照，使同一人在不同光照条件下提取的特征保持稳定。接着进行轻微锐化——补偿摄像头常见的轻微模糊，增强面部细节。最后通过深度学习识别模型从预处理后的人脸中提取高维特征向量，并归一化为单位长度。

特征提取采用不同的策略适配不同场景。考勤签到场景下，系统重新检测人脸以确保特征提取的目标区域与质量检查的结果一致。合照识别场景下，系统直接使用已有的人脸检测框提取特征，避免二次检测可能产生的位置偏移。

### 4.4 身份比对

身份比对将当前人脸的特征向量与人脸库中所有已录入学生的特征向量逐一计算相似度。相似度的度量采用余弦相似度——计算两个向量的夹角余弦值，完全同向为 1，完全反向为 -1。余弦相似度不受向量模长影响，天然适合衡量归一化后的特征向量。

相似度原始值域为 -1 到 1，系统将其线性映射到 0 到 1 的范围，方便理解和设定阈值。所有候选人按相似度降序排列，取最高分作为匹配结果。最终判定需要同时满足匹配到学生且相似度超过预设阈值两个条件，缺一不可。

对于考勤场景，每位学生人脸库中可能存有多张照片（多次录入），系统将这些照片的特征做加权平均后存储为一个代表特征。这种合并策略既保留了多次录入带来的多样性信息，又避免了数据库膨胀和比对时的重复计算。对于合照识别场景，系统采用向量化批量计算——一次矩阵运算得出当前人脸与全库所有特征的相似度。

### 4.5 考勤数据记录与去重

考勤记录无论成功或失败都写入数据库——失败记录对于发现人脸库质量问题、潜在攻击行为同样具有价值。每条记录包含完整的时间戳、课程名称、识别结果状态、置信度、活体检测评分、情绪类型及失败原因。

同一学生同一天对同一课程的多次签到采用覆盖更新策略。连续自动捕捉模式下，同一学生可能被反复识别到，每次写入新记录会造成数据膨胀。系统在每次签到成功时查询当天同课程是否已有成功记录，有则更新（覆盖置信度、活体分、情绪、时间、照片），无则新建。失败记录不受影响，每次都独立写入，因为它们可能是安全事件，不应被覆盖。

### 4.6 统计数据生成

统计数据通过对原始记录的聚合查询实时生成，不依赖预计算或缓存。课程列表和日期列表从考勤记录中动态提取不同值，这使得系统无需维护单独的课程表或教学计划——教师每次输入什么课程名，统计就能按什么课程名聚合。

课次编号按每个课程的历史签到日期升序自动编号。同一课程不同日期对应不同课次，系统收集该课程的所有不重复日期后按时间排序，取序号即为课次。每课次的签到人数通过条件计数实时计算。出席率通过签到人数与学生总数的比值计算得出。

### 4.7 数据导出

系统支持考勤明细和统计报表两种维度的 Excel 导出。导出功能基于服务器端生成，通过开源电子表格库构建工作簿，设置列宽、表头样式后以二进制流返回。前端接收后触发浏览器下载。

考勤明细导出受当前筛选条件影响——如果当前按课程和日期筛选了考勤记录，导出的即为筛选后的数据。统计报表导出覆盖指定课程的全部出勤汇总。合照识别导出包含活动参与名单，附带每位学生的累计活动参与次数。

## 五、考勤与合照识别

### 5.1 考勤签到

考勤签到是系统的核心业务链路，串联了从图像采集到结果记录的完整流程，是最能体现系统综合能力的功能。

**交互设计**。系统支持两种签到模式切换。手动捕捉模式下，用户主动点击按钮触发单次签到——摄像头截取当前画面、执行完整管线、返回结果。连续自动捕捉模式下，系统以固定间隔自动截取画面并执行管线的各个阶段——人脸检测、活体验证、身份比对，签到成功后进入短暂冷却期（避免同一人反复触发），冷却结束后继续下一轮检测。连续模式适合全班排队签到——学生依次走过摄像头前，无需触碰任何按钮，门槛降到最低。

**视觉反馈**。签到界面向用户提供多层次的状态反馈。视频画面实时绘制人脸检测框——主脸（面积最大者）和次要人脸用不同颜色区分。传统的人脸引导框提示用户将面部对准画面中央。状态栏显示当前处于哪个阶段（等待人脸、验证动作、识别中、签到结果）。步骤指示器展示整体进度条。这种设计让签到过程透明化，用户随时知道"签到进行到哪一步了"。

**活体检测管线**。活体检测是防欺骗的核心防线，分为反欺骗分析和动作验证两个子阶段。

反欺骗分析对当前画面进行多维度评分。深度学习反欺骗模型判断整体真实性，脉搏波检测利用绿色通道信号的频域分析提取心跳信息（真人有周期性脉搏、照片没有），摩尔纹检测通过二维频谱的异常峰值识别屏幕翻拍的干涉条纹，光流分析区分真人面部的非刚体局部运动与照片的整体刚体平移，纹理和频域统计辅助判断。各维度按预设权重综合评分，低于阈值则判定为欺骗。

动作验证是第二道防线。后端从预设动作池中随机抽取多个动作指令返回给前端，前端逐项提示用户完成。微笑检测利用面部特征检测器验证笑容的出现，转头检测通过鼻尖位置相对于基线的偏移判断头部是否真实转动，张嘴检测通过嘴部区域的高度比例判断是否张嘴。每个动作有独立的验证逻辑——关键是要检测"从基线到当前帧发生了真实变化"而非仅仅检测"当前帧是否符合"。全部动作通过后活体检测完成。

**身份比对与判定**。活体检测通过后，进入人脸特征提取和全库比对环节。最终判定需同时满足三个条件：匹配到学生、置信度超过预设阈值、活体检测通过。任一条件不满足即判定为失败，并给出具体的失败原因——是"未匹配到人脸库中的学生"还是"活体检测未通过"。

### 5.2 合照识别

合照识别面向教师组织活动后的批量考勤需求，与单人签到的技术栈共享但处理逻辑独立。

教师在前端页面拖拽或选择合照图片上传，预览确认后点击开始识别。后端进行全人脸检测——检测画面中所有人脸，不限制数量。每张人脸独立提取特征，然后与人脸库全量比对。与单人考勤不同，合照识别采用向量化批量计算提升比对效率。同一学生在同一张照片中可能被多次拍到，系统通过去重确保每个学生只被计入一次。

识别成功的自动写入活动参与记录和情绪记录，无需教师额外操作。识别失败的人脸也不丢弃——系统返回所有人脸框的坐标，前端在原图上叠加标注：已识别学生用绿色边框并显示姓名，未识别人脸用红色边框。这为教师提供了直观的识别效果一览，未识别的情况可能是底库中缺少该学生数据或人脸质量不足。

系统同步统计每位已识别学生的历史累计活动参与次数（按不重复活动名计数），在识别结果中展示，并在导出报表中包含此项。教师完成一次识别即可看到"谁参加了多少次活动"的完整信息。

## 六、系统安全能力

系统安全不能仅局限于活体检测——活体检测解决的是"考勤数据真实性"这一特定业务安全问题，而作为一个完整的信息系统，还需要应对常规的网络安全威胁。安全设计遵循纵深防御原则，从多个层面构建防护体系。

### 6.1 业务安全：活体检测与防欺骗

这是课程评分中权重最高的单项能力，也是系统在"内容安全"主题下的核心体现。需求是抵御三种常见攻击：打印照片（成本最低）、屏幕翻拍（最常见）、预录视频（最隐蔽）。

**防照片攻击**。打印或显示在屏幕上的照片缺乏真人的生理特征。脉搏波检测利用心跳导致的面部皮肤颜色微变化——真人面部绿色通道信号中存在与心跳同步的周期性波动，照片则只有环境光的低频变化或传感器的随机噪声。纹理分析检测打印介质或屏幕像素网格的微观特征，通过局部二值模式的统计分布区分皮肤和纸张/屏幕。深度学习反欺骗模型通过大量样本训练学习真人与伪造品在高维特征空间中的决策边界。

**防屏幕翻拍**。屏幕翻拍比打印照片更难检测，因为手机屏幕亮度高、色彩饱和，感官上更接近真人。系统针对屏幕翻拍的特性增加专项防御。摩尔纹检测利用摄像头传感器像素网格与屏幕像素网格叠加产生干涉条纹的物理现象——这种干涉在图像的二维频谱中表现为异常的周期性高频峰值。频域综合分析从更宏观的层面检视频谱衰减模式——屏幕翻拍经过了"屏幕显示→摄像头拍摄"的二次采样，频谱在特定频段会出现不自然的衰减或增强。

**防视频伪造与动作挑战**。预录视频攻击是最难通过被动检测应对的手段——视频本身就是真人拍摄的，包含心跳信号、真实的纹理和正常的频谱特征。针对这类攻击，系统引入交互式随机动作挑战。

核心设计思想是：攻击者无法预知系统会要求什么动作，也无法让预录的视频实时响应随机指令。后端从动作池随机抽取动作序列，前端逐项提示用户在摄像头前完成。每个动作有独立的验证逻辑，关键是验证"从基线到当前帧发生了符合要求的变化"而非仅仅"当前帧看起来像某种表情"。

光流时序分析为动作挑战提供补充。真人的面部运动是局部的、非刚体的——不同区域（眼睛、嘴、脸颊）以不同方向、不同幅度运动。而照片或屏幕的整体晃动产生的是全局的、一致的刚体平移。这种光流模式差异可以被精确捕捉。

### 6.2 输入安全：注入防护与参数校验

所有外部输入都是不可信的，系统在多个层面进行输入安全防护。

**SQL 注入防护**。系统使用 ORM 框架构建数据库操作，参数化的查询语句从架构层面阻止了 SQL 注入的可能——用户输入始终作为数据参数传递，不会与 SQL 语句拼接。少数需要动态条件组合的查询场景，通过 ORM 提供的条件构建器安全地组装查询。

**文件上传安全**。文件上传是系统最主要的外部数据入口，也是最容易成为攻击向量的接口。系统对所有上传图片进行多重校验：MIME 类型必须位于白名单内（不允许可执行文件、脚本文件等危险类型伪装为图片）、文件大小受上限限制、图片内容必须能被成功解码为合法的图像数据。校验失败直接拒绝，并返回明确的错误说明。

**参数校验**。所有接口的输入参数通过类型化 Schema 进行校验——路径参数、查询参数、请求体都有明确的类型定义和约束。不符合预期的参数在进入业务逻辑之前被拦截并返回校验错误，避免因异常输入导致的后端错误或未预期行为。

**跨站脚本防护**。前端框架默认对模板中的变量进行 HTML 转义，从框架层面防止了反射型 XSS。后端返回的用户数据和错误消息不包含可执行的脚本内容。

### 6.3 认证与授权安全

**密码安全**。用户密码不以明文存储，而是通过 bcrypt 算法进行单向哈希后再存入数据库。bcrypt 内置盐值和可配置的计算成本，能有效抵御彩虹表攻击和暴力破解。密码验证时从不取出原哈希值进行比较，而是通过算法重新计算后比对。

**令牌安全**。认证令牌采用 JWT 标准，包含用户身份和角色信息，具有过期时效。令牌通过 HMAC 签名保护，防止篡改。前端存储令牌后，每次 API 请求通过拦截器自动附加，无需在每个接口中手动处理。

**权限分层校验**。权限控制不是单一的"登录检查"，而是分三层执行：认证层确认"你是谁"（JWT 解析身份），授权层确认"你能做什么"（角色依赖注入），数据层确保"你只能看你的"（查询时自动限定数据范围）。学生查询考勤记录和情绪记录时，后端自动过滤为非本人数据不可见。

### 6.4 错误处理与异常防护

统一的异常处理策略确保系统在任何异常情况下都能给出合理的响应，而非暴露内部实现细节或直接崩溃。

接口层面的异常被统一捕获和处理。预期的业务异常（如人脸检测失败、识别不通过）返回明确的错误码和用户可理解的错误说明。非预期的系统异常被兜底捕获，记录错误日志，向前端返回通用的错误提示而非堆栈信息——避免向外部暴露内部实现细节。

前端的网络请求具备错误拦截和静默失败能力。对于关键操作（如签到提交），错误会弹出提示引导用户重试。对于后台持续执行的检测类请求（如实时的视频人脸检测），失败时静默处理，不打断用户正在进行的操作。不可恢复的错误（如摄像头权限被拒绝）给出明确的引导提示。

### 6.5 数据完整性

考勤记录作为教学管理的基础数据，需要保证完整性和可追溯性。记录一旦写入不做物理删除（关键记录在删除前会弹出二次确认），字段信息完整（时间、课程、学生、置信度、活体分、情绪、失败原因全部写入）。照片单独持久化保存，不因记录的更新而丢失历史图像。

人脸特征的合并更新采用加权平均策略——每次新增人脸样本时，旧特征与新特征按样本数量加权合并后重新归一化。这种策略既保留了历史录入的信息，又逐步优化了特征的代表性，避免单次低质量录入污染整体特征质量。

## 八、超出评分范畴的系统能力

以下能力虽不直接对应评分条目，但反映了系统作为完整软件工程项目的综合质量。

**实时视频人脸检测**：在签到视频流上以每秒约 4 帧的频率进行实时人脸检测，在画面上绘制彩色人脸框。这替代了早期基于图像亮度方差的粗糙估计，提供了真实的视觉反馈。检测通过独立的轻量接口完成，不干扰签到主流程。

**图像格式兼容**：超出评分对"支持 JPG/PNG"的基本要求，系统接受 WebP、AVIF、BMP、TIFF 等多种图像格式。对于非主流格式，后端自动进行格式转换后处理，而非直接拒绝。这降低了用户的操作门槛——不需要关心照片是什么格式。

**考勤去重**：同一学生同课程同一天的多次签到自动覆盖更新而非重复写入。这解决了连续自动捕捉模式下的数据膨胀问题，同时保持了记录的最新性。

**课次统计**：从考勤记录中动态计算每门课程的课次编号和每课次签到人数，在考勤记录页面实时展示。无需维护排课表或教学计划。

**合照人脸可视化**：识别完成后在原图上叠加人脸检测框和标注，已识别和未识别用不同颜色区分。为教师提供直观的识别质量反馈。

**学生自助服务**：学生可自行登录系统，查看个人信息，通过拍照或上传录入人脸照片，修改登录密码。减少了教师的人工管理负担。

**批量人脸导入**：通过文件名规则（学号-姓名-班级-性别）解析信息，一次性完成学生创建、账号生成、特征提取的全流程。大幅降低了初期人脸库建设的操作成本。

**前端组件复用**：人脸标注框、学生详情弹窗等组件独立封装，在多页面间复用，减少了重复代码。
