> 对照 `request.md` 评分标准，逐项分析当前实现方案、技术优势、存在缺陷与整改措施。

---

## 一、架构要求（10分）

### 1.1 BS分层架构清晰，各层职责明确（3分）

**当前实现方案**

```
frontend/                          backend/
├── src/                           ├── app/
│   ├── views/      → 页面视图      │   ├── api/        → 路由层（参数校验、响应封装）
│   ├── api/        → HTTP 封装     │   ├── services/   → 业务逻辑层（人脸/活体/情绪/导出）
│   ├── stores/     → 状态管理      │   ├── models/     → ORM 数据模型
│   ├── router/     → 路由守卫      │   ├── schemas/    → Pydantic 请求/响应模型
│   └── layout/     → 布局框架      │   └── core/       → 配置/数据库/安全基础设施
```

- **前端**：Vue 3（Composition API + `<script setup>`）+ Vite 构建 + Element Plus UI + Pinia 状态管理 + Vue Router 4 路由守卫
- **后端**：FastAPI 异步框架 → `api/` 路由层（Depends 依赖注入） → `services/` 业务逻辑（单例模式） → `models/` SQLAlchemy ORM → `core/` 基础设施（config / database / security）
- **数据库**：SQLite + SQLAlchemy 2.x declarative mapping，5张表通过 ForeignKey 关联

**优势**

| 方面 | 说明 |
|---|---|
| 前后端分离 | RESTful API 通过 JSON 通信，前端 Vite 开发服务器代理到后端，部署时可独立托管 |
| 依赖注入 | FastAPI `Depends` 实现认证、数据库会话的声明式注入，代码简洁、可测试性强 |
| 单例服务 | `face_service` / `liveness_service` / `emotion_service` 以模块级单例暴露，ONNX 模型只加载一次，避免重复初始化 |
| 类型安全 | Python 侧 `Mapped` 类型注解 + Pydantic v2 校验；前端 JS 无类型但 API 封装集中在 `modules.js`，调用链路清晰 |

**缺陷**

| 方面 | 说明 |
|---|---|
| 无 Service 层抽象 | 路由函数直接操作 SQLAlchemy Session（如 `attendance.py:75-78` 在路由中拼接查询），业务逻辑分散在 api 层，复杂场景下难以复用和测试 |
| 单例不可测试 | `face_service = FaceService()` 模块级单例，单元测试时无法 mock 替换 |
| 缺少全局异常中间件 | FastAPI 未注册 `exception_handler`，HTTPException 之外的异常（如数据库连接失败）会直接暴露出 500 内部细节 |
| 前端无 TypeScript | Vue 3 支持 TS 但项目使用纯 JS，API 响应类型无编译期校验，字段拼写错误要到运行时才发现 |

**整改方案**

1. 在 `api/` 和 `services/` 之间增加 **repository 层**（或至少在 api 中将复杂查询提取为独立函数），避免路由函数直接拼接 SQL
2. 将服务单例改为 **可注入的依赖**（FastAPI `Depends` + 工厂函数），便于测试时替换
3. 添加全局异常处理器，统一返回 `{ "detail": "...", "code": "ERROR_CODE" }` 格式
4. 可选：前端迁移到 TypeScript，利用类型系统减少运行时错误

4. 各视图补充 `el-empty` 空状态占位

### 1.3 后端API接口规范，可实现核心逻辑交互（2分）

**当前实现方案**

| 方面 | 实现 |
|---|---|
| 路由聚合 | `api/__init__.py` 使用 `APIRouter().include_router()` 汇总 6 个子模块，统一 `/api` 前缀 |
| 参数校验 | Pydantic v2 `BaseModel` 校验请求体；FastAPI `Query`/`File`/`Depends` 自动校验并生成 OpenAPI 文档 |
| 认证 | OAuth2 Password Bearer → JWT 签发/解码 (`python-jose`) → `get_current_user` 依赖注入 |
| 权限 | `require_teacher` 依赖检查 `user.role == "teacher"`，拒绝非教师操作 |
| 响应格式 | 列表接口直接返回 list，单对象返回 Pydantic model，导出返回 `StreamingResponse` |
| 错误处理 | 各路由中 `raise HTTPException(status_code=400/401/403/404, detail="...")` |

**优势**

| 方面 | 说明 |
|---|---|
| 自动文档 | FastAPI 自动生成 Swagger UI (`/docs`) 和 ReDoc (`/redoc`)，参数和响应模型完整 |
| 认证体系完整 | JWT 8小时过期 + bcrypt 密码哈希 + 教师/学生角色区分 |
| 批量操作设计合理 | 批量导入逐一处理、失败不中断（`try/except` 单张继续），返回 `imported/failed` 明细 |
| 级联删除 | 删除学生时自动清理关联的考勤/情绪/活动/用户数据 + 磁盘人脸照片 |

**缺陷**

| 方面 | 说明 |
|---|---|
| 响应格式不统一 | 列表接口直接返回 `[...]`，无分页元数据（total/page/pageSize），与标准 REST 分页规范不一致 |
| 缺少请求限流 | 无 rate limiting，`/check-in` 等接口可能被高频调用（连续捕捉模式每 800ms 一次） |
| 无 CORS 配置 | FastAPI 默认允许所有来源，生产环境存在安全隐患 |
| JWT secret 硬编码 | `config.py` 中 `secret_key = "change-me-in-production"`，未从环境变量或密钥管理服务读取 |
| 无日志系统 | 只有 `print` 或异常堆栈，缺少结构化日志（请求ID、耗时、用户信息） |

**整改方案**

1. 统一分页响应格式：`{ "items": [...], "total": N, "page": N, "page_size": N }`
2. 添加 `slowapi` 或自实现限流中间件，对 `/check-in` 接口限制每秒请求数
3. 配置 CORS 白名单（`fastapi.middleware.cors.CORSMiddleware`）
4. `secret_key` 改为 `os.environ.get("SECRET_KEY", default)` 从环境变量读取
5. 接入 `loguru` 或标准 `logging`，记录请求日志

### 1.4 数据库设计合理，数据表关联正确（2分）

**当前实现方案 — 5张表 ER 关系**

```
users ────┐                          students ─────┐
  id PK    │ FK student_id (nullable)   id PK       │ FK
  username │                            student_no   │
  role     │                            name         │
           │                            class_name   │
           │                            gender       │
           │                            face_encoding│
           │                            face_image   │
           └────────────────────────────────────────┘
                                      │              │
                ┌─────────────────────┼──────────────┼──────────────────┐
                │ FK                  │ FK           │ FK               │ FK
          attendance_records    emotion_records  activity_participation
            id PK                 id PK           id PK
            student_id            student_id      student_id
            timestamp             timestamp       activity_name
            status                emotion_type    activity_date
            confidence            confidence      recognized
            liveness_score        source          confidence
            emotion_type
            course_name
            message
```

**优势**

| 方面 | 说明 |
|---|---|
| 合适的索引策略 | `student_no`、`name`、`class_name` 高频查询字段已建索引；`timestamp`、`emotion_type`、`activity_name` 等统计字段有索引 |
| 级联处理在前端 | 批量删除学生时手动清理关联数据（`students.py:214-228`），比数据库级联更可控（可记录清理数量） |
| 编码存储高效 | `face_encoding` 使用 `LargeBinary` 直接存 numpy float32 的原始字节，无序列化开销，读取即用 |
| 活动参与表预留扩展 | `ActivityParticipation` 有 `recognized` 和 `confidence` 字段，可区分人工确认和自动识别 |

**缺陷**

| 方面 | 说明 |
|---|---|
| `ActivityParticipation` 未被写入 | 表定义完整但 `group_photo.py` 未向此表插入数据，合照识别的参与记录只写入 `AttendanceRecord`，导致 activity stats 接口返回空 |
| `User.student_id` 无唯一约束 | 一个学生可被多个账号绑定，可能造成权限混乱 |
| `face_sample_count` 与实际文件数可能不一致 | 合并特征时 `+1` 计数（`students.py:79`），但若上传失败后重试可能导致计数偏大 |
| 缺少软删除 | 学生和记录都是硬删除，误操作后无法恢复 |
| SQLite 并发写瓶颈 | SQLite 单写锁，多人同时签到时可能出现 `database is locked` 错误 |

**整改方案**

1. **关键修复**：`group_photo.py` 识别成功时写入 `ActivityParticipation` 表
2. `users.student_id` 添加 `unique=True` 约束
3. 可选：生产环境迁移到 PostgreSQL（FastAPI + SQLAlchemy 切换只需改连接串）

## 二、核心功能1：基础考勤（25分）

### 2.1 前端 — 摄像头调用正常、人脸采集流畅（3分）

**当前实现方案**

`frontend/src/views/Attendance.vue:205-231`

```javascript
async function startCamera() {
  if (cameraStream) { /* 清除旧流 */ }
  cameraStream = await navigator.mediaDevices.getUserMedia({
    video: { width: { ideal: 1280 }, height: { ideal: 720 }, facingMode: 'user' },
    audio: false,
  })
  videoRef.value.srcObject = cameraStream
  cameraActive.value = true
}
```

- 使用浏览器原生 `getUserMedia` API，请求 720p 前置摄像头
- canvas 绘制视频帧 → `toBlob('image/jpeg', 0.95)` 输出 JPEG
- 组件销毁时 `onBeforeUnmount` 释放摄像头流

**优势**

- 原生 API，无第三方依赖，兼容所有主流浏览器
- 720p 分辨率平衡了清晰度和传输体积
- JPEG 质量 0.95 保证识别精度
- 切换摄像头时先释放旧流再获取新流，避免资源泄漏

**缺陷**

| 方面 | 说明 |
|---|---|
| 无设备选择 | 笔记本有多个摄像头时只能使用默认前置，无法手动选择 |
| 无权限引导 | 用户拒绝摄像头权限时只有 `ElMessage.error`，未提供浏览器设置指引 |
| 无对焦/曝光提示 | 部分摄像头需要几秒自动对焦，期间采集的图像可能模糊 |

**整改方案**

1. 使用 `navigator.mediaDevices.enumerateDevices()` 列出视频设备，提供切换下拉框
2. 权限拒绝时给出更详细的引导（如"请在浏览器设置中允许摄像头权限"的图文说明）
3. 启动摄像头后等待 500ms 再做首次采集，给自动对焦留时间

### 2.2 前端 — 支持手动/自动捕捉并显示采集状态（3分）

**当前实现方案**

| 模式 | 代码位置 | 流程 |
|---|---|---|
| 手动拍照 | `captureAndSubmit()` :287 | 点击"完成考勤"→拍照→提交→显示结果 |
| 3秒自动 | `autoCapture()` :325 | 点击"3秒自动拍照"→SVG环形倒计时3→2→1→自动拍照提交 |
| 连续自动捕捉 | `toggleContinuousCapture()` :341 | 每800ms检测人脸→自动拍照→提交→显示结果→冷却3秒→循环 |

状态显示：
- 步骤条 `el-steps` 实时标记当前阶段（摄像头→活体→识别→结果）
- 连续模式：状态条 `captureStatus` 显示"等待人脸→检测中→识别中→签到成功/失败→冷却中"
- 人脸引导框：`faceDetected` 为 true 时变绿色 + "人脸已检测"

**优势**

- 三种模式覆盖不同场景：手动精细控制、倒计时定时、连续自动批量
- 状态反馈层次清晰：步骤条（宏观）+ 状态条（实时）+ 人脸框（视觉引导）
- 连续模式有 3 秒冷却机制，避免同一人重复签到
- 倒计时使用 SVG 环形进度条（`stroke-dasharray/dashoffset` 动画），视觉效果精致

**缺陷**

| 方面 | 说明 |
|---|---|
| 前端人脸检测是伪实现 | `faceDetected` 仅基于像素亮度方差（`Attendance.vue:260-264`），任意明亮物体都可能被误判为"人脸已检测" |
| 连续模式没有失败次数限制 | 识别一直失败会无限循环，浪费带宽和计算资源 |
| 倒计时期间无取消按钮 | 3秒倒计时开始后用户无法中断 |

**整改方案**

1. 前端人脸检测改为调用后端轻量接口或使用 face-api.js（TensorFlow.js），或保持现状但明确标注"画面质量检测"
2. 连续模式增加最大失败次数（如连续 5 次失败自动停止），或仅在 faceDetected 状态下才开始检测
3. 倒计时期间增加取消按钮

### 2.5 后端 — 活体检测可抵御照片、视频欺骗（10分）⭐⭐⭐

> 这是评分占比最高的单项（10分），也是当前实现最大的短板。

**当前实现方案**

`liveness_service.py:161-227` — 7 维度综合评分：

| 维度 | 方法 | 权重(有DeepFace) | 权重(无DeepFace) |
|---|---|---|---|
| 清晰度 | Laplacian 方差 | 5% | 20% |

**步骤 3**：`liveness_service.py` 新增 `verify_action()` 方法

```python
def verify_action(self, image: np.ndarray, action: str,
                  face_box: tuple[int,int,int,int]) -> tuple[bool, str]:
    """验证用户是否完成指定活体动作。
    
    YuNet 检测返回的 15 维向量：
    [x, y, w, h, x_re, y_re, x_le, y_le, x_n, y_n, x_rc, y_rc, x_lc, y_lc, angle]
    其中关键点坐标相对于原图。
    """
    if action == "smile":
        return self._verify_smile(image, face_box)
    elif action in ("turn_left", "turn_right"):
        return self._verify_head_turn(image, face_box, action)
    elif action == "open_mouth":
        return self._verify_mouth_open(image, face_box)
    return True, ""

def _verify_smile(self, image, face_box) -> tuple[bool, str]:
    x, y, w, h = face_box
    mouth_roi = image[y+int(h*0.65):y+h, x+int(w*0.2):x+int(w*0.8)]
    smiles = self.smile_cascade.detectMultiScale(
        cv2.cvtColor(mouth_roi, cv2.COLOR_BGR2GRAY),
        scaleFactor=1.3, minNeighbors=10, minSize=(18, 10))
    if len(smiles) > 0:
        return True, ""
    return False, "未完成指定动作：请微笑后完成考勤"

def _verify_head_turn(self, image, face_box, action) -> tuple[bool, str]:
    """使用 YuNet 关键点：鼻尖 x 坐标相对脸框位置判断转头方向。"""
    sface_rows = self._detect_sface_rows(image)  # 需暴露 YuNet 结果
    if not sface_rows:
        return False, "无法检测人脸关键点，请正对摄像头"
    # sface_rows 的 15 维格式中，鼻尖 x = face[8], 脸框 x = face[0], 脸宽 = face[2]
    face = sface_rows[0]
    nose_x, face_x, face_w = face[8], face[0], face[2]
    ratio = (nose_x - face_x) / face_w
    if action == "turn_left" and ratio > 0.55:
        return True, ""
    if action == "turn_right" and ratio < 0.45:
        return True, ""
    action_text = "向左转头" if action == "turn_left" else "向右转头"
    return False, f"未完成指定动作：请{action_text}后完成考勤"
```

| 考勤签到 | `attendance.py:74` `emotion_service.analyze(image, faces[0])` |
| 合照识别 | `group_photo.py:50` `emotion_service.analyze(image, box)` |

每次调用时：
1. 裁剪人脸区域（40% padding）
2. 尝试 DeepFace.analyze(actions=["emotion"])
3. 失败时回退到 OpenCV 启发式分类

**优势**

- 统一的 `analyze()` 接口，两种场景复用
- 裁剪后再分析，减少背景干扰
- 情绪结果同步存入 `EmotionRecord`，可溯源（source 标记 attendance/group_photo）

**缺陷**

| 方面 | 说明 |
|---|---|
| 考勤和合照的 analyze 是同步阻塞调用 | DeepFace 加载模型+推理可能耗时 1-3 秒，直接阻塞签到响应 |
| 没有异步化 | FastAPI 支持 `async`，但 `emotion_service.analyze()` 是同步方法，可能阻塞事件循环 |
| 合照中逐人脸串行分析 | 50 人合照 × 1-3 秒/人 = 50-150 秒响应时间，用户体验差 |

**整改方案**

1. 情绪分析放入线程池执行（`asyncio.to_thread` 或 `run_in_executor`），避免阻塞主事件循环
2. 合照识别中多人脸情绪分析并行化

#### 4.1.2 可完成情绪分类（3分）

**当前实现方案**

`emotion_service.py:40-98` — 双模分类：

**模式1：DeepFace**（优先）
- 7 类：happy, sad, angry, surprised, fearful, disgusted, neutral
- 返回主导情绪 + 置信度（百分比/100）

**模式2：OpenCV 启发式**（回退）
- 同样 7 类
- 特征：灰度均值、对比度、边缘密度、Sobel梯度、mouth亮度、smile cascade、eye cascade
- 线性加权得分 → softmax 归一化 → 选最大概率

**优势**

- DeepFace 作为主力，准确率有保障
- 回退方案确保 DeepFace 不可用时仍能工作
- 启发式分类利用了 smile/eye cascade 等实际可检测的特征（微笑→happy、眼睛多→surprised）
- softmax 归一化保证了概率分布合理

**缺陷**

| 方面 | 说明 |
|---|---|
| 启发式准确率低 | 手工规则+线性加权对复杂情绪（fearful/disgusted）几乎无法区分，实测 neutral 倾向严重 |
| 无情绪持续性分析 | 单次签到只记录一个情绪快照，无法反映真实心理状态 |
| DeepFace 情绪模型较重 | 每次 analyze 都要加载模型（虽然 DeepFace 内部有缓存），首次调用慢 |

**整改方案**

1. 启发式回退时降低 `fearful`、`disgusted` 的权重（这两种在静态图片中极难判断），提高 neutral 覆盖
2. 可选：使用更轻量的情绪模型（如 `fer` 库的 ONNX 模型）替代 DeepFace

## 五、核心功能4：系统安全（18分）

### 5.1 抗攻击能力（15分）

> 此项目的核心改进项，详见 [2.5 活体检测整改方案](#25-后端--活体检测可抵御照片视频欺骗10分)。

#### 5.1.1 抗照片伪造（7分）

**当前状态**：⚠️ 不足，存在严重安全漏洞

| 现有防护层 | 有效性评估 |
|---|---|
| Laplacian 清晰度检测 | 低 — 高质量打印照片同样清晰 |
| FFT 频域分析 | 中 — 对屏幕翻拍（摩尔纹）有效，对高质量打印照片效果有限 |
| LBP 纹理分析 | 中 — 对普通打印纸有效，对光面照片纸效果下降 |
| 色彩多样性 | 低 — 高质量打印色彩同样丰富 |
| DeepFace MiniFASNet | 高 — 但当前是可选的，未被强制使用 |
| Challenge 动作验证 | **无** — 返回了动作但完全不验证 |

**整改方案**

1. **强制 DeepFace**：`requirements.txt` 添加 `deepface`，`liveness_service.py` 删除 try/except 的 fallback 路径
2. **提高阈值**：有 DeepFace → 0.55，无 DeepFace → 0.45
3. **验证动作**：必须完成指定动作才算活体通过
4. **错误消息分级**：
   - 翻拍检测："检测到照片/屏幕翻拍，请使用真实人脸签到"
   - 动作未完成："未完成指定动作：请XXX后完成考勤"

#### 5.1.2 抗视频伪造（8分）

**当前状态**：⚠️ 同样不足

| 现有防护层 | 有效性评估 |
|---|---|
| FFT 频域分析 | 中 — 屏幕翻拍的视频会有像素级周期性条纹 |
| LBP 纹理 | 低 — 高清视频播放的纹理与真人接近 |
| DeepFace | 高 — 但当前可选 |
| 动作挑战 | **无** — 视频播放同样可以不做动作 |

**整改方案**

视频攻击的特征与照片不同：
1. **频域分析加强**：视频屏幕翻拍的 FFT 尖峰比照片更明显（刷新率条纹），可提高 FFT 权重
2. **动作挑战是关键防线**：视频可以录下人做动作的过程，但挑战是**随机的**（4 种动作中选 1 种），攻击者无法预知该播放哪种动作的视频。**随机挑战 + 严格验证**是抵御视频重放攻击的核心手段
3. **时间戳校验**：要求 challenge 在 15 秒内完成（当前已有 `expires_in: 15`），超过则需刷新动作

### 5.2 权限与异常防护（3分）

**当前实现方案**

| 层面 | 实现 |
|---|---|
| 认证 | JWT (python-jose, HS256, 8小时过期) + bcrypt 密码哈希 |
| 角色区分 | `users.role` ∈ {teacher, student} |
| 教师权限 | `require_teacher` 依赖注入，保护学生CRUD/合照识别等操作 |
| 学生隔离 | `attendance.py:37-38` 学生只能查询自己的考勤记录 |
| 密码安全 | bcrypt + salt，数据库不存明文 |

**优势**

| 方面 | 说明 |
|---|---|
| JWT 无状态 | 服务端不需维护 session，水平扩展友好 |
| 依赖注入简洁 | 一行 `Depends(require_teacher)` 即可保护整个端点 |
| 学生数据隔离 | 在查询层过滤（`filter(AttendanceRecord.student_id == user.student_id)`），不会越权 |

**缺陷**

| 方面 | 说明 |
|---|---|
| 无 refresh token | JWT 8小时过期后需重新登录 |
| 无账号锁定 | 暴力破解无防护（无登录失败次数限制） |
| 无操作日志 | 无法审计谁在何时做了何种操作 |
| `secret_key` 硬编码 | `"change-me-in-production"` 直接写在代码中 |
| 师生共用同一登录入口 | 无区分，登录后由 role 字段决定权限 |

**整改方案**

1. `secret_key` 从环境变量读取：`os.environ.get("SECRET_KEY", os.urandom(32).hex())`
2. 添加登录失败次数限制（5次错误锁定15分钟）
3. 添加操作审计日志（至少在删除/批量导入时记录）
4. 可选：实现 refresh token 机制
