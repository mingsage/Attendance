# 评分点实现报告

## 一、架构要求（10 分）

### ① BS 分层架构清晰，各层职责明确（3 分）

**设计思路**

BS 架构的本质是把"界面"和"计算"解耦。人脸检测、特征提取、活体验证、数据库操作都是计算密集型或 IO 密集型任务，必须放在服务端——浏览器无法运行 ONNX 推理、无法访问数据库、无法保证模型文件的安全。而摄像头调用、画面渲染、用户交互是浏览器的天然优势——服务端无法直接操作摄像头、无法实时绘制 UI。

因此分层的第一原则是：**AI 算在服务端，UI 跑在浏览器端**，二者只通过 HTTP 交换数据。

在此基础上，前后端各自内部再分层。分层的核心不是"分几个文件夹"，而是**每个模块只做一件事，且只能调用它下一层的模块**——视图层不直接操作数据库，服务层不直接渲染 HTML。

**前端层次结构**

前端从外到内分为四层。视图层是最外层，每个页面是一个独立的 Vue 组件，只负责模板渲染和用户事件绑定，不包含任何业务逻辑。状态层通过 Pinia 管理登录用户信息和 JWT 令牌，所有需要用户身份的组件从 store 读取而非 props 层层传递。路由层控制页面跳转，在跳转前内置角色校验——学生访问教师页面的 URL 会被自动重定向。API 层封装所有后端接口调用，统一管理请求前缀、JWT 附加、错误拦截。

视图层中的签到页面是前端最复杂的组件，它管理着摄像头流、检测定时器、连续捕捉循环、活体动作状态机、结果渲染等多个状态，但仍遵循"只展示、不算"的原则——它从不运行人脸检测算法，只把画面发送给后端。

**后端层次结构**

后端从外到内也分为四层。路由层接收 HTTP 请求，做参数校验和权限检查，然后委托给服务层处理，自己不做算法计算。服务层是 AI 能力的封装体——人脸服务、活体服务、情绪服务各司其职，互相独立，可单独替换。模型层定义数据库表结构和字段约束，通过 ORM 框架将 Python 对象映射为 SQL 操作。核心层管理全局配置、数据库连接池、JWT 密钥等基础设施。

路由层中的签到接口是后端最长的函数，但它本质上是"编排"——依次调用人脸服务检测、质量检查、特征提取、活体服务分析、情绪服务分析、人脸服务比对，最后写入数据库。它没有自己实现任何一个算法。

**依赖方向**

```
视图层 → API 层 → (HTTP) → 路由层 → 服务层 → 模型层
                                              ↕
                                          核心层（配置/连接）
```

每一层只依赖其正下方的一层。视图层不知道数据库的存在，服务层不知道 HTTP 请求的格式，模型层不知道外面的业务逻辑。

**代码组织**

前端代码目录中，views、stores、router、api、components、layout 各自独立。一个页面组件（如签到页）导入 api 模块发请求、导入 store 取用户信息、使用 components 中的复用组件——这三类依赖被明确分隔。后端代码目录中，api（路由）、services（服务）、models（模型）、core（核心）各自独立。路由文件导入服务模块和模型模块，从不反向导入。

### ② 前端适配主流浏览器，界面简洁、交互友好（3 分）

**设计思路**

浏览器兼容性的前提是使用标准 API。摄像头调用使用 W3C 标准的 MediaDevices API（getUserMedia），Chrome、Firefox、Edge 均原生支持。UI 组件使用 Element Plus 框架，其底层已处理各浏览器的 CSS 兼容性和表单行为差异。不做浏览器特定的 hack，不依赖非标准特性。

界面简洁的核心是"信息分层"——最重要的信息最大最醒目，次要信息小且低调，辅助信息默认隐藏或折叠。考勤签到页的核心信息是"签到结果"（成功/失败 + 学生姓名），这是最大的元素；置信度和活体分是辅助验证信息，用小号字体展示在次要位置；步骤指示器是过程信息，用紧凑的横条展示。

交互友好的关键是"即时反馈"——用户的每个操作都要有可感知的响应。签到按钮点击后立即显示加载状态，连续捕捉模式用状态条实时显示当前阶段，人脸检测框在视频上实时绘制，告诉用户"系统正在看你的脸"。

**关键技术实现**

摄像头权限处理。浏览器可能因为用户拒绝、设备被占用、无摄像头等原因无法打开摄像头。前端在调用 getUserMedia 失败时捕获异常并弹出明确的错误提示（"摄像头启动失败，请检查浏览器权限"），而非显示空白画面或崩溃。

状态反馈体系。签到页面通过步骤指示器、状态栏、结果面板三个区域提供层次化反馈。步骤指示器用简洁的四步条（摄像头→活体动作→人脸识别→结果）告诉用户整体进度。状态栏在连续捕捉模式下实时显示"等待人脸…"、"请微笑"、"识别中…"、"签到成功：张三"。结果面板在签到完成后展示完整的识别结果。

响应式布局。页面使用弹性布局和最小宽度约束，在常见的笔记本屏幕（1366×768）到桌面显示器（1920×1080）之间都能正常显示。工具栏按钮在宽度不足时自动换行。表格列宽设置了合适的最小宽度，内容不会挤压变形。

**交互细节**

- 表格行悬停时高亮，点击可查看照片或详情
- 操作按钮在行内展示，无需先选中再点顶部按钮
- 批量删除前弹出二次确认对话框，防止误操作
- 导出按钮带下拉菜单区分"导出明细"和"导出报表"
- 搜索结果为空时显示友好的空状态而非空白表格

### ③ 后端 API 接口规范，可实现核心逻辑交互（2 分）

**设计思路**

API 设计遵循 RESTful 风格的两条核心原则。第一，URL 以资源名词为中心——`/students` 管理学生、`/attendance/records` 管理考勤记录、`/group-photo` 处理合照识别。第二，HTTP 方法表达操作语义——GET 查询、POST 创建、PUT 更新、DELETE 删除。

文件上传和导出是特殊的操作接口。签到需要上传摄像头截取的图片，人脸录入需要上传照片——这些接口使用 multipart/form-data 编码。Excel 导出接口返回二进制流，前端通过创建 Blob URL 触发浏览器下载。

**接口清单结构**

考勤模块围绕签到动作和记录管理展开。`POST /check-in` 执行一次完整签到（接收图片和课程名，返回签到结果）。`GET /records` 查询历史签到记录（支持关键词、状态、课程名筛选）。`POST /detect` 提供轻量的人脸检测（只检测不识别，用于前端视频实时画框）。`/liveness-challenge` 获取随机活体动作。`/verify-action` 验证单个动作是否完成。`/records/{id}` 和 `/records/batch-delete` 管理记录的删除。

学生管理模块提供标准的资源 CRUD。`GET /students` 列表查询（支持关键词筛选）。`POST /students` 创建学生。`PUT /students/{id}` 更新。`DELETE /students/{id}` 删除。`POST /students/{id}/face` 录入单张人脸。`POST /students/faces/batch` 批量导入人脸。

合照识别模块仅一个核心接口 `POST /group-photo/recognize`，在内部完成检测、识别、记录写入的全流程。

统计模块提供多个 GET 接口：仪表盘汇总、情绪分布、活动统计、考勤出勤统计、课程列表、日期列表及导出接口。

**统一规范**

所有接口通过 Pydantic Schema 定义输入和输出的数据结构。响应对象有明确的类型定义（字段名、类型、是否可选），前端可以通过这些定义生成 TypeScript 类型或直接查阅 OpenAPI 自动文档。错误响应统一包含 HTTP 状态码（400 客户端错误、403 权限拒绝、404 资源不存在、500 服务端错误）和 detail 字段说明错误原因。

JWT 认证通过 FastAPI 的依赖注入实现。需要登录的接口声明 `Depends(get_current_user)` 依赖，框架自动解析请求头中的 Bearer 令牌并注入当前用户对象。需要教师权限的接口额外声明 `Depends(require_teacher)` 依赖，该依赖在校验通过后什么也不做，在校验失败时抛出 403 异常。

### ④ 数据库设计合理，数据表关联正确（2 分）

**设计思路**

数据库以"学生"为中心实体构建。考勤记录、情绪记录、活动参与记录都关联到学生，用户表通过外键关联到学生。这种设计的理由是——系统的所有业务操作（签到、识别、统计）最终都围绕"哪个学生"展开，以学生为锚点的关联模型最自然。

人脸特征不是单独建向量表，而是作为学生表的一个字段存储。在班级规模（数十到数百人）下，单次全库遍历比对的性能完全可接受，引入额外的向量存储反而增加了数据一致性的维护负担。特征的存储格式统一为二进制浮点数组，入库前和出库时有统一的编解码函数。

**表结构**

学生表存储学号（唯一索引）、姓名、班级、性别等基本信息，以及人脸特征的二进制字段和人脸样本计数。样本计数记录该学生合并了多少张照片的特征——单张就是 1，多次录入就是 N。

用户表存储用户名（即学号）、bcrypt 密码哈希、角色（teacher/student）、关联的学生 ID。一个学生最多对应一个用户账号（学生 ID 唯一约束）。教师账号不关联学生。

考勤记录表是数据量最大的表，每次签到写入一条记录。字段覆盖了一次签到的全部信息——关联的学生 ID、时间戳、课程名称、状态（成功/失败）、置信度、活体检测评分、情绪类型、失败原因消息、签到照片路径。时间戳和课程名有联合索引支持统计查询。

情绪记录表独立于考勤记录表存储。因为情绪数据有两个来源（考勤签到和合照识别），且有自己的查询需求（按来源统计、按情绪类型聚合）。通过来源字段区分数据的出处，通过时间戳和学生 ID 与原始记录关联。

活动参与表记录合照识别产生的结果——活动名称、日期、学生 ID、是否被识别、置信度。同一学生参加不同活动会写入多条记录，这是后续活动次数统计的数据基础。

**关联关系**

用户到学生是一对一（或零对一，教师不关联学生）。考勤记录到学生是多对一（一个学生有多条签到记录）。情绪记录到学生是多对一（一个学生有多条情绪记录）。活动参与到学生是多对一（一个学生参加多次活动）。

所有外键关系在 ORM 模型中定义，查询时通过关系加载（joinedload）一次性取出关联数据，避免 N+1 查询问题。

## 二、基础考勤 — 前端（12 分）

### ① 摄像头调用正常、人脸采集流畅（3 分）

**如何实现**

浏览器通过标准 MediaDevices API 请求用户授权摄像头。系统指定了 1280×720 的理想分辨率和 user-facing（前置摄像头）的偏好。这个分辨率在保证人脸细节清晰的同时，不会因数据量过大导致传输延迟。

采集流程高度简洁：打开摄像头后，视频画面实时显示在页面上；用户（或自动检测）触发捕捉时，通过 Canvas 从视频流中绘制当前帧，压缩为 JPEG 格式（95% 质量），转换为 File 对象后通过 FormData 上传。整个采集过程对用户透明——不需要选择分辨率、不需要手动保存文件、不需要知道 JPEG 是什么。

摄像头失败处理是采集流畅性的重要组成部分。getUserMedia 可能因为多种原因失败——权限被拒绝、设备被其他应用占用、没有物理摄像头、浏览器不支持。每种失败都会给用户明确的提示，而非空白画面或 console 报错。切换摄像头时旧的视频流被正确释放，避免资源泄漏。

**关键代码逻辑**

摄像头请求指定 1280×720、前置摄像头、无音频。成功后视频流绑定到 video 元素，同时启动实时人脸检测定时器。手动捕捉从 video 元素创建 Canvas、绘制当前帧、生成 Blob、包装为 File 对象。压缩质量 95% 在文件大小和画质之间取平衡——大约 100-300KB，普通网络环境下 1-2 秒即可上传完成。

### ② 支持手动/自动捕捉并显示采集状态（3 分）

**如何实现**

系统提供两种捕捉模式，对应不同的课堂场景。手动捕捉：学生主动走到摄像头前点击按钮，系统截取当前帧完成签到。适合零星补签或需要学生自主操作的场景。连续自动捕捉：摄像头持续运行，每隔固定间隔（约半秒）自动截取画面并启动签到管线。当画面中检测到人脸时自动进入活体验证和签到流程，签到成功后进入短暂冷却期（3 秒），冷却结束后继续检测下一个。适合全班排队签到——学生依次走过摄像头前无需触碰任何按钮。

状态反馈分三个层次。步骤指示器用四步横条（摄像头→活体动作→人脸识别→结果）展示一次签到的整体进度，当前步骤高亮。状态栏在连续捕捉模式下实时显示文本——"等待人脸…"、"请微笑"、"识别中…"、"签到成功：张三"。这比传统的 loading 转圈提供了更有意义的信息。结果面板在签到完成后出现，用结构化描述列表展示学号、姓名、班级、置信度、情绪、活体分、时间等完整信息。成功时绿色提示，失败时红色提示。

**关键代码逻辑**

连续捕捉循环通过 JavaScript 定时器实现，每半秒执行一次。循环体首先检查多个前置条件——连续模式是否开启、是否处于冷却期、上一轮是否处理完毕、距上次失败是否足够久。条件满足后从视频中截帧，分两阶段执行：阶段一是活体动作串行验证（逐一尝试当前要求的动作直到全部通过），阶段二是完整签到提交。签到成功后启动冷却定时器（倒数 3 秒），冷却期间暂停检测。

### ③ 能实时渲染后端返回的学生信息（3 分）

**如何实现**

签到提交后，后端返回包含完整识别结果的 JSON 对象。前端拿到结果后立即在页面上渲染——不是弹一个 alert 或 toast，而是以结构化描述列表的形式展示。渲染内容包括：学号、姓名、性别、班级、置信度（带颜色指示——高于阈值绿色否则红色）、情绪（emoji + 中文文本）、活体检测评分、课程名称、签到时间、人脸库照片数量。

成功和失败两种状态都有完整的信息展示。成功时展示识别到的学生身份和置信度；失败时展示失败原因（"未检测到人脸"、"人脸质量不足"、"活体检测未通过"、"匹配分数低于阈值"等），帮助学生或教师判断问题所在。

渲染有渐入动画，不会突然闪现。上一个结果在新结果出现前自动清除，避免信息混淆。

**关键代码逻辑**

签到接口返回一个 CheckInResponse 对象，包含 success 布尔值和 record 对象（完整的学生信息、时间戳、置信度等）。前端将 result 赋值后，Vue 的响应式系统自动触发模板重新渲染。结果面板通过过渡动画组件包裹，有渐入效果。情绪字段通过映射表转换为 emoji + 文本的组合展示。

连续捕捉模式下，状态栏中的文本会立即更新为签到结果——"签到成功：张三"或失败原因，学生不需要去看详细的结果面板就能知道签到结果。

### ④ 支持筛选查询考勤记录（3 分）

**如何实现**

考勤记录页面提供多维筛选能力。课程下拉框列出历史所有签到过的课程名（从记录中动态提取）。选择课程后，日期下拉框列出该课程历史上签到过的所有日期（按时间升序）。此外还有姓名/学号关键词搜索框和签到状态（成功/失败）下拉筛选。

筛选条件的组合是"AND"关系——同时选课程和日期，返回的是该课程该日期的记录。筛选条件变化时自动重新查询，无需手动点击搜索按钮（关键词搜索除外）。清除筛选条件同样自动刷新。

课程和日期都选择后，页面上方出现统计概览栏——应到人数（该班级全部学生）、实到人数（该课程该日期签到成功的学生）、缺勤人数、到课率百分比。缺勤学生以标签形式列出姓名和学号。这个统计来自独立的统计接口，不受考勤明细表的筛选条件影响。

每门课程在考勤记录中还附带课次信息——系统收集该课程所有不重复的历史签到日期，按时间升序排列，每条记录所在的日期序号即为"课次 N"，并显示该课次的签到总人数。这为教师提供了额外的上下文——不是孤立的"某条签到记录"，而是"某课程第几次课的签到"。

**关键代码逻辑**

课程列表接口查询考勤记录表中所有不重复的课程名。日期列表接口查询指定课程的所有不重复日期（按天截取时间戳）。课次编号在查询考勤记录后计算：收集所有涉及的课程名，对每个课程查询所有不重复日期并按升序排列，每条记录根据其日期在列表中的位置确定编号（索引+1）。每课次签到人数通过查询当天同课程记录的成功条数得出。

数据导出支持当前筛选条件的 Excel 导出，以及课程维度的统计报表导出。

## 二、基础考勤 — 后端（13 分）

### ⑤ 活体检测功能正常，可抵御照片、视频欺骗（10 分）

**设计思路**

这是课程评分中单点分值最高的一项（10 分），也是系统技术含量最集中的模块。活体检测的目标是区分"一个真人在摄像头前"和"一张照片/一段视频被摄像头拍摄"。

单靠一种检测手段不可靠——深度学习反欺骗模型可能被高质量伪造绕过；脉搏波检测在弱光下信噪比低；摩尔纹检测依赖屏幕的显示特性。但如果多种手段叠加，攻击者需要同时绕过所有维度。因此系统的核心策略是**多维证据综合评分**，而非依赖单一银弹。

同时，针对最隐蔽的预录视频攻击，纯被动检测手段（无论多少个维度）都无法可靠区分"一个真人在动"和"一段真人视频在播放"。唯一的解法是引入**交互式随机动作挑战**——攻击者无法预知系统会要求什么动作，无法让录好的视频实时响应随机指令。

**多维反欺骗体系**

脉搏波检测利用了一个生物学事实：真人的面部皮肤颜色会随心跳周期发生微小的周期性变化，在绿色通道中尤为明显。系统采集连续帧的绿色通道均值信号，对其进行频域分析，检测心跳频段（约 0.8-3Hz）内是否存在显著的周期性峰值。照片和屏幕显示没有心跳——它们的颜色变化只来自环境光，频谱是平坦的或低频的。这一维度被赋予一票否决的权重——连基本心跳信号都没有的"人脸"直接判定为欺骗。

摩尔纹检测针对屏幕翻拍。当摄像头拍摄手机或电脑屏幕时，摄像头传感器的像素网格与屏幕的像素网格叠加会产生干涉条纹（摩尔纹）。这种条纹在图像的二维频谱中表现为异常的高频峰值——真实人脸没有这种规则的周期性纹理。系统对图像进行二维傅里叶变换，检测频谱中是否存在异常的周期性峰值。

光流时序分析区分真人面部运动和照片整体晃动。真人面部运动是局部、非刚体的——眼睛、嘴巴、脸颊以不同方向、不同幅度运动。而手持照片晃动是全局、刚体的——所有像素几乎在同一方向上移动相同的距离。系统计算连续帧之间的光流场，分析运动矢量的方向一致性和空间分布。如果所有像素的运动方向高度一致（低熵），则高度怀疑是照片平移。

深度学习反欺骗模型作为辅助判断。模型在大量真人和伪造样本上训练，学习了两者在高维特征空间中的决策边界。作为补充，系统还分析图像的纹理统计（局部二值模式——纸张和屏幕与皮肤的纹理分布不同）、频域衰减特性（屏幕翻拍在高频段有不自然的衰减）、色彩分布、清晰度、亮度等辅助特征。

各维度结果按预设权重汇总为综合评分——核心维度（脉搏波、反欺骗模型）权重高，辅助维度权重低。评分低于动态阈值则判定为欺骗。

**随机动作挑战**

互动式验证从根本上杜绝预录视频攻击。系统后端从动作池（微笑、转头、张嘴）中随机抽取多个动作，前端逐项提示用户在摄像头前完成。每个动作有独立的验证逻辑。

微笑检测利用面部特征检测器——在嘴部区域扫描是否存在与笑容模式匹配的特征。转头检测更加精妙——它不判断当前帧是不是"侧脸"，而是先记录用户面朝前方时的鼻尖位置作为基线，然后对比后续帧中鼻尖是否发生了真实的横向位移。如果鼻尖从基线位置偏移超过阈值，判定转头动作完成。张嘴检测通过嘴部区域的高度与面部高度的比例变化判断——张嘴比例超过阈值则判定动作完成。

关键是验证"变化的发生"而非"状态的匹配"。攻击者的照片即使恰好拍到笑脸或侧脸，也无法产生"从正脸到侧脸"的鼻尖位移过程。

**技术实现要点**

脉搏波检测需要积累约 30 帧（约 2-3 秒）才能提取可靠的频域信号——单帧无法完成。这要求前端在活体检测阶段保持画面稳定。摩尔纹检测和光流分析同样需要连续帧。系统将检测设计为"窗口式"——每次分析基于一个滑动窗口内的多帧数据，而非孤立的单帧。

### ⑥ 班级人脸库可批量导入、单个添加/删除/修改（2 分）

**如何实现**

人脸库管理围绕学生档案 CRUD 和照片录入展开。单个学生的管理提供标准的增删改查操作——新增学生时填写学号、姓名、性别、班级，保存后可立即录入人脸照片。编辑学生时修改基本信息。删除学生需经过二次确认，同时清理其考勤记录、情绪记录、活动记录和登录账号。

人脸照片录入支持两种方式——拍照录入（开启摄像头后拍摄）和上传照片（从本地文件选择）。拍照录入在弹窗内打开摄像头，人脸引导框提示用户对准位置，点击拍照后自动完成特征提取和质量检查，通过后写入特征库。上传模式流线型体验——选择文件后直接上传，无需中间步骤。

批量导入人脸是提高人脸库建设效率的核心功能。教师准备好多张学生照片，按"学号-姓名-班级-性别.jpg"的命名规则命名后，一次性选择全部上传。系统逐张解析文件名提取学生信息，检测照片中的人脸并提取特征，在数据库中找到或创建对应的学生记录，合并或写入人脸特征，自动创建登录账号（如果尚不存在）。导入完成后展示结果统计——成功导入多少张、失败多少张及失败原因（文件名缺少学号、未检测到人脸等）。

**关键实现逻辑**

批量导入中每张照片的处理是独立的——一张失败不影响其他。特征合并采用加权平均——如果学生已有人脸特征（旧特征源于 N 张照片），新特征与旧特征按 N:1 的权重合并后重新归一化。这保证了多次录入的特征逐步优化而非简单覆盖。

### ⑦ 可记录考勤数据并支持 Excel 格式导出（1 分）

**如何实现**

考勤记录无论签到成功还是失败都写入数据库。记录包含完整的时间戳、课程名称、学生信息、置信度、活体评分、情绪类型、状态和失败原因。成功时额外写入情绪记录（关联到考勤记录）。

数据导出分两种模式。考勤明细导出——将当前筛选条件下的考勤记录导出为 Excel，包含学号、姓名、班级、时间、课程、状态、置信度、情绪等列。统计报表导出——选定课程后，导出该课程全部日期的每学生出勤统计。

导出在前端触发——点击导出按钮后前端发起请求，后端生成 Excel 二进制流返回，前端创建 Blob URL 并触发浏览器下载。导出文件按日期命名方便存档。

## 三、合照识别（9 分）

### ① 支持前端上传合照，后端可检测并批量识别人脸（3 分）

**设计思路**

合照识别面向场景与单人考勤不同——教师已经有一张包含多人的合影照片，需要从中认出每一位学生。流程分为"上传→检测→逐一比对→返回结果"四个阶段。

前端上传区域支持拖拽和点击两种方式，上传后预览照片方便确认。识别按钮触发后端调用，完成后在预览照片上叠加人脸检测框（已识别绿色、未识别红色），给了教师直观的识别效果反馈。识别结果以表格展示，包含学号、姓名、班级、置信度（进度条）、情绪。

**技术实现**

后端接收图片后首先进行全人脸检测——使用与考勤签到相同的检测器，但检测全部人脸而非取最大。每张检测到的人脸独立提取特征。特征提取不重新进行人脸检测，而是使用已有的检测框信息——因为重新检测可能导致检测结果不一致（偏移、遗漏），造成识别错位。

每个人脸的特征与人脸库中全部已录入学生的特征进行比对。合照场景采用向量化批量计算——一次矩阵点积运算即可得出当前人脸与全库的相似度排序，比逐个循环计算效率更高。

相似度超过阈值且该学生尚未在本张合照中被识别过的，判定为识别成功。去重机制确保同一学生在合照中被拍到多次也只计一次。识别成功的学生自动写入活动参与记录和情绪记录，无需额外的保存操作。

识别结果中返回所有人脸框的坐标（包括未识别的），前端据此在预览图上叠加标注框。识别成功的显示绿色边框和姓名标签，未识别的显示红色边框。

**关键实现逻辑**

人脸框坐标以像素值返回，前端根据原图尺寸与显示尺寸的比例计算缩放，用绝对定位的 div 绘制矩形框。这一标注功能已封装为独立的 Vue 组件（FaceOverlay），可复用于其他需要展示人脸框的页面。

---

### ② 合照识别准确率 ≥85%，可处理 10-50 人，生成含学号姓名的活动参与名单（3 分）

**如何实现**

识别准确率的核心保障来自三个方面。特征提取模型本身在公开人脸识别基准上具有良好的准确率——在标准姿态、正常光照条件下，同一人的特征向量在高维空间中高度聚集。光照归一化预处理补偿了合照中常见的不均匀光照问题——CLAHE（对比度受限自适应直方图均衡化）增强暗部细节。特征提取前的几何对齐消除头部姿态偏差。

合照场景的匹配阈值设定比单人考勤略低——合照中的人脸通常更小、角度更随机，适当放宽阈值可以提升召回率，减少漏识别。每人库中有多张照片时，特征经过加权平均后更具代表性，进一步提升了匹配的鲁棒性。

生成的活动参与名单包含学号、姓名、班级、置信度、情绪（emoji + 文本）和累计活动参与次数。这份名单即是评分要求的"含学号姓名的名单"。

---

### ③ 可自动记录每位学生活动参与次数（2 分）

**如何实现**

每次合照识别成功后，系统自动在活动参与表中写入一条记录——包含活动名称（由教师在界面上输入，默认为"班级活动"）、活动日期（当天日期）、学生 ID、是否被识别、置信度。这不需要教师做任何额外操作——识别完成即写入完成。

活动参与次数的统计通过数据库聚合查询实现——按学生 ID 统计不重复的活动名称数量。这个数量在当前识别结果中实时展示（"活动次数：N 次"），让教师直观看到每个学生参与了多少次班级活动。历史累计次数在后续每次识别中都会更新。

合照识别产生的活动参与记录同时也是考勤记录详情页中"参与活动"展开列表的数据源——点击一条考勤记录的展开按钮，可以看到该学生参与过哪些活动及每次的置信度。

**关键实现逻辑**

活动次数在识别接口中实时计算——commit 后查询本次识别涉及的所有学生的不重复活动名称数量，附加到返回结果中。前端在表格中用标签形式展示。Excel 导出同样包含活动次数列，数据来源于相同的聚合查询逻辑。

---

### ④ 能生成统计数字、表格、柱状图等形式的统计报表（1 分）

**如何实现**

合照识别的统计报表分布在多个页面。合照识别页面本身在识别完成后展示摘要卡片——检测到的人脸数、成功识别数、未识别数，以大字数字 + 标签文字展示，一目了然。识别结果以表格形式列出所有被识别的学生及其置信度和情绪。

仪表盘（总览页面）中的"活动参与"指标卡统计了活动参与表的总记录数，是一个宏观的活动频次指标。考勤记录表格的展开行展示每位学生参与的活动列表。

合照识别页面的导出按钮可将当前活动（或全部活动）的参与名单导出为 Excel 文件，包含活动名称、日期、学号、姓名、班级、置信度、情绪、累计活动次数等完整字段。

## 四、情绪分析（8 分）

### ① 考勤、合照识别过程中可同步提取面部特征（2 分）

**设计思路**

情绪分析被设计为签到和合照识别管线中的一个同步环节——不是独立的功能入口，而是在主流程的自然节点上附带执行。这满足评分要求中"同步提取"的含义：用户在做考勤或合照识别时，情绪分析自动发生，无需额外的操作步骤或专门的"情绪分析"功能入口。

同步的实现方式是函数调用层面的——签到函数在人脸检测和特征提取完成后立即调用情绪分析函数，情绪分析函数与活体检测并行或先后执行（不相互阻塞），然后将结果一并写入数据库。

**关键实现逻辑**

签到接口中，在人脸比对之前调用情绪分析函数，传入原始图像和人脸框坐标。情绪分析函数内部裁剪人脸区域后进行分析。合照识别接口中，对每个识别成功的人脸同样调用情绪分析函数。两个场景共享同一个情绪分析函数，保证分类标准和数据格式的一致性。

---

### ② 可完成情绪分类（3 分）

**如何实现**

情绪分类的目标是将人脸表情归类为 7 种基本情绪之一。系统采用决策链策略——主力引擎 + 兜底方案，确保在任何部署环境下都能输出合理的结果。

主力引擎使用 DeepFace 框架中的情绪识别模型。该模型基于深度学习，在标准表情数据集上训练，能区分七种基本情绪。引擎返回每种情绪的概率分布和主导情绪类型。由于不同引擎返回的标签名称可能有细微差异（如 "fear" vs "fearful"），系统在返回结果前做统一的标签归一化映射。

当 DeepFace 不可用（如依赖未安装）或分析失败时，系统降级为基于 OpenCV 的启发式分类——利用微笑检测器、眼部检测器、梯度分析等手段，对面部特征进行统计并映射到情绪类别。这个降级方案的准确率不如深度学习模型，但保证了系统的可用性——在任何部署条件下都不会因为缺少一个依赖而完全无法工作。

情绪分析在裁剪后的人脸区域上运行——只分析人脸部分，排除背景干扰。这比在全图上分析更准确且更高效。

---

### ③ 后端可记录学号、姓名、识别时间、情绪类型的完整数据（2 分）

**如何实现**

情绪记录独立存储在情绪记录表中。每条记录包含：关联的学生 ID（通过外键可获取学号、姓名等完整信息）、分析时间戳（精确到秒）、情绪类型（7 种之一）、置信度、数据来源（考勤签到 "attendance" 或合照识别 "group_photo"）。

来源标记是重要的设计——它区分了情绪数据是来自单人签到（通常是正面近距离、表情自然）还是合照（可能是侧面远距离、表情各异）。在统计时可以按来源过滤或分组，提供更精细的分析视角。

学生 ID 的外键关联使得查询时可以一次性加载学生的学号、姓名等信息，在统计页面展示完整的"谁、什么时候、什么情绪"的记录。

**关键实现逻辑**

考勤接口中，签到成功时额外执行一条情绪记录的写入操作。合照识别接口中，每个识别成功的人脸自动写入情绪记录。写入操作与主记录在同一个数据库事务中，保证数据一致性。前端统计页通过情绪记录查询接口获取数据，按来源区分展示。

### ④ 前端支持查看统计结果（1 分）

**如何实现**

情绪统计页面分为上下两部分。上半部分是 ECharts 饼图——每种情绪用不同颜色标识，图块大小反映该情绪在全部记录中的占比。鼠标悬停显示具体次数和百分比。下半部分是明细表格——列出每条情绪记录的时间、学号、姓名、情绪类型（emoji + 文本）、置信度和数据来源。

教师端查看时不区分数据来源——考勤和合照产生的情绪记录混合统计。学生端查看时仅显示自己考勤签到产生的情绪记录。数据显示的兼容处理：前端映射表同时包含了归一化后的标签（fearful）和原始标签（fear），确保不同时期产生的数据都能正确展示。

## 五、系统安全（18 分）

### ① 活体检测抗照片伪造攻击效果良好（7 分）

**设计思路**

照片伪造是最低成本、最常见的攻击手段——学生用手机拍一张他人的照片，对着摄像头即可尝试冒充签到。对抗这种攻击的核心策略是发现"这不是一个活人"的多维证据。

系统综合运用脉搏波检测（照片无心跳信号）、深度学习反欺骗模型（区分真人与打印品/屏幕的高维特征）、纹理分析（纸张纤维和屏幕像素网格的微观纹理不同于皮肤）、摩尔纹检测（摄像头拍摄屏幕时产生的干涉条纹），以及色度、清晰度、亮度、频域衰减等辅助判断。这些维度覆盖了不同伪造方式——打印在纸上的、显示在手机屏幕上的、高分辨率平板的。

各维度不是简单加和——脉搏波检测作为一票否决项：如果连基本的心跳信号都没有，其他维度无需再看。深度学习模型权重最高——它经过大量样本训练，是区分真伪最可靠的单点手段。纹理和摩尔纹次之——它们针对特定媒介。其他维度作为辅助——在核心维度信号模糊时帮助判定。

**关键实现逻辑**

活体检测分析函数接收连续多帧图像（而非单帧），计算各维度指标后加权综合。返回综合评分和是否通过判定的布尔值。阈值是可配置的——可以根据实际测试效果调整，平衡误拒率和漏过率。

---

### ② 活体检测抗视频伪造攻击效果良好，杜绝虚假考勤（8 分）

**设计思路**

这是评分中分值最高的单项能力（8 分），也是最难的技术挑战。视频伪造比照片伪造更难检测——因为视频本身就是真人拍摄的，画面中有心跳信号（被拍摄的真人）、有自然的纹理和频谱特征、没有摩尔纹。纯被动检测手段对视频攻击的效果十分有限。

系统的核心对策是**随机动作挑战**——既然被动手段无法区分"真人"和"真人的视频"，那就引入交互——让系统要求用户做出不可预测的即时动作，而攻击者的预录视频无法实时响应。

动作挑战的设计有几个关键点。动作必须是**随机选择**的——攻击者无法预知系统会要求哪个动作，就无法提前录制对应的视频。动作必须是从**正常状态到目标状态**的过程——仅检测"当前帧是否有笑容"不够（照片可能拍到笑容），要检测"从无笑容到有笑容的变化"。动作必须在**限时内**完成——防止攻击者现场摆拍或切换视频。

**技术实现逻辑**

后端从微笑、转头、张嘴三个动作中随机抽取多个组成挑战序列。每个动作有独立的验证函数——微笑用 Haar 微笑检测器，转头用鼻尖位置偏移（先记录基线再对比），张嘴用嘴部区域比例变化。

转头检测的设计尤为精巧——它不是简单地判断"人脸是否侧向"，而是先记录用户初始时刻的鼻尖在脸部框中的相对位置（基线），然后等待后续帧中鼻尖位置发生显著的横向位移。只有当位移超过阈值时才判定转头完成。这杜绝了"照片本身就是侧脸照"的攻击。

动作验证是**有状态**的——后端的验证会话记录了当前用户被分配了哪些动作、当前进行到第几个、转头的基线鼻位是多少。这保证了验证的连续性和一致性——不能跳过动作、不能在不同动作间切换攻击素材。

**整体防御矩阵**

综合来看，系统对照片伪造（被动检测即可覆盖）和视频伪造（被动+交互双覆盖）形成了有梯度的防御。攻击者需要同时绕过脉搏波、反欺骗模型、摩尔纹、纹理分析等被动防线，还要实时响应随机的交互式动作挑战——这在实践中几乎不可能。

### ③ 实现教师、学生账号权限区分（3 分）

**如何实现教师/学生权限区分**

权限区分的核心设计原则是**最小权限原则**——每种角色只能访问其职责范围内的功能和数据。教师的职责是管理考勤流程和学生数据，拥有全局查看和管理权限；学生的职责是参与签到和维护个人信息，权限限于个人范围。

**教师权限**

教师拥有系统的全部操作和数据访问权限。在考勤签到页面，教师可以开关活体检测、管理活体动作。在考勤记录页面，教师可以查看全部学生的记录（不受学生 ID 限制）、批量删除记录、导出全量数据。在学生管理页面，教师可以进行学生档案的增删改查、录入和批量导入人脸。在合照识别页面，教师可以上传合照并查看识别结果。在情绪统计页面，教师可以看到全部学生的情绪数据（不分来源）并查看签到照片。

**学生权限**

学生权限限于个人操作范围。学生可以使用考勤签到功能完成签到（与教师共用同一页面但无法开关活体检测）。学生可以查看考勤记录页面（可以看到所有记录但无法删除他人数据）。学生有独立的"我的信息"页面——查看学籍信息、上传或拍照录入自己的人脸照片、修改登录密码。学生无法访问总览仪表盘（教师专属）、学生管理页、合照识别页。

**权限校验的三层实现**

认证层保障"你是谁"。登录时后端验证密码，成功后签发 JWT 令牌。令牌包含用户 ID、用户名和角色（teacher/student）。前端将令牌存储在 Pinia 的 auth store 中，通过 Axios 拦截器自动附加到每次请求的 Authorization 头。后端通过依赖注入解析令牌并实例化当前用户对象。

路由层保障"你能去哪"。前端路由表为每条路由声明允许的角色（meta.role）。全局导航守卫在每次页面跳转时读取目标路由的角色要求，与当前用户角色比对。学生试图访问 `/students`（学生管理）或 `/group-photo`（合照识别）时自动重定向到学生首页 `/my-profile`。教师访问学生专属页面 `/my-profile` 同理被拦截。

接口层保障"你能做什么"。后端为教师专属接口添加了角色依赖——在路由函数签名中声明 `Depends(require_teacher)`。这个依赖检查当前用户角色是否为 teacher，不是则抛出 HTTP 403 异常。数据层面对学生施加数据范围限制——学生查询考勤记录时后端自动添加"仅限本人 student_id"的过滤条件；学生查询情绪记录时自动限定来源为 attendance 的数据。

三层防护的意义在于——不能依赖任何单一层次的安全。前端路由守卫可以被绕过（浏览器开发者工具），但接口层仍然会拒绝未授权的请求。JWT 令牌可以被篡改（如果有密钥泄露），但 bcrypt 哈希的密码本身不受影响。

**技术实现要点**

学生用户的创建与批量导入人脸联动——系统解析文件名中的学号，在创建或更新学生记录的同时调用账号创建函数。如果该学号尚未有登录账号，自动创建（用户名=学号，默认密码=123456，角色=student）。学生首次登录后可以在"我的信息"页面修改密码。这避免了教师需要手动为每个学生创建账号的繁琐操作。
