## 理解确认（待用户核实）

### 考勤页面 — 学生视角

**关于"刷新动作"**

当前页面有一个"刷新动作"按钮（`Attendance.vue:82-84`），点击后重新请求 `/liveness-challenge` 获取一个新的随机动作。用户表示不理解其存在意义——如果动作每次考勤自动下发，为什么要手动刷新？

**两种考勤模式**（替换当前的"手动/3秒/连续"三模式）

| 模式 | 用户描述 | 理解 |
|---|---|---|
| 模式一：3秒拍照 | 选择此模式→视频展示框等待3秒→获取那一帧→后台识别→视频定格在该帧→页面展示识别结果 | 类似当前的 `autoCapture()`，但区别是：拍完后视频定格，结果直接覆盖在画面上 |
| 模式二：实时识别 | 每隔几帧送后台识别→实时返回信息→视频画面上标注人脸框→如果识别到"当前登录账号的人脸"，就**允许**提交考勤 | 不同于当前的"连续自动捕捉"（自动提交），新模式是：实时识别但不自动提交，识别到本人后才允许手动提交 |

**实时人脸标注**

- 当前：视频上覆盖一个固定的 SVG 椭圆引导框 + "人脸已检测/请对准" 文字
- 改为：实时将后台返回的人脸框绘制在视频上（canvas overlay），标注出画面中的人脸位置；如果识别成功，特别高亮标注出当前登录学生的人脸

**活体和情绪**

- 用户认可当前实现方式，作为安全手段保留，不需要大改交互

### 两种模式的行为差异

| | 3秒拍照模式 | 实时识别模式 |
|---|---|---|
| 视频状态 | 实时预览→3秒倒计时→定格 | 始终实时预览 |
| 后台调用时机 | 倒计时结束后一帧 | 每隔几帧持续调用 |
| 人脸标注 | 定格后展示 | 实时绘框+高亮 |
| 提交方式 | 自动提交 | 识别到本人后手动提交 |
| 适用场景 | 单人快速签到 | 持续监控/多人场景 |

### 与当前实现的差异

| 当前 | 改为 |
|---|---|
| 三模式：手动/3秒/连续自动 | 两模式：3秒拍照/实时识别 |
| 连续模式自动提交 | 实时模式识别到本人后**手动**提交 |
| 固定椭圆引导框 | 实时人脸框标注+本人高亮 |
| "刷新动作"按钮 | 用户质疑，需确认是否保留 |

## 理解确认

考勤流程从"识别即提交"改为"识别→确认→提交"：

**改前**：摄像头捕捉 → 后端识别成功 → 自动写入考勤记录 → 返回结果

**改后**：
```
摄像头实时识别 ──→ 识别到本人 ──→ 页面提示"已识别" ──→ 用户点击[提交考勤]
                                                              │
                                                  POST /check-in (正式提交)
                                                              │
                                                      写入 AttendanceRecord
                                                              │
                                                      返回"考勤成功"
```

**关键变化**：

| | 改前 | 改后 |
|---|---|---|
| 提交方式 | 识别成功即自动写入 | 识别成功只是"允许提交"，用户手动点按钮才写入 |
| 多次提交 | 每次识别成功都写一条记录 | 识别期间只提交一次，避免重复 |
| 后端职责拆分 | check-in 一个接口做识别+活体+提交 | 识别接口（轻量，实时跑）+ 提交接口（正式写入） |

这意味着后端需要拆成两个接口：
- **识别接口**：`POST /attendance/recognize` — 只做人脸检测+比对，返回"是否识别到本人"和置信度，不写数据库
- **提交接口**：`POST /attendance/check-in` — 携带当前帧+活体检测，正式写入考勤记录

## 理解确认（待用户核实）

### 学生端：查看自己的考勤

| 项目 | 说明 |
|---|---|
| 筛选维度 | 日期 / 课程/活动 / 状态（已签/缺签） |
| 展示形式 | 表格，行首显示统计数（如"共 15 条，已签 12，缺签 3"） |
| "应该签但没签" | 学生需要知道：教师发布了哪些考勤活动，哪些自己没签到 |

关键前提：学生要看到"该去但没去"的，系统必须知道"哪些活动该去"——这就需要**教师发布考勤活动+指定参与学生/群组**，否则系统只能展示"已签到的"，无法计算"缺签的"。

### 教师端：群组管理（权限补充）

> 教师选定某些学生/已有班级成为一个群组，然后针对这些群组发布关于某活动/课程的考勤活动

| 项目 | 说明 |
|---|---|
| 群组 | 教师可以创建群组，从学生库中选人（或按班级筛选批量加入） |
| 考勤活动 | 教师发布活动时，指定：课程名 / 日期 / 课次 / 参与群组 |
| 作用 | 定义了"谁该来"→ 缺签统计才有依据 |

### 教师端：查看考勤统计

| 项目 | 说明 |
|---|---|
| 筛选维度 | 日期 / 班级 / 活动/课程 / 学号 / 姓名 |
| 展示形式 | 表格 + 统计数据 + 导出 Excel |
| 表格列 | 学号 / 姓名 / 班级 / 参与次数（已签） / 未参与次数（缺签） / 到课率 |
| 同名课程累计 | 如"数学课"共 10 次，某学生参与 8 次，缺 2 次 → 显示 8/10 |

### 与当前实现的差异

| | 当前 | 改为 |
|---|---|---|
| 学生看自己考勤 | 后端过滤但前端无学生视角 | 学生端独立页面：已签/缺签表格+统计数 |
| "缺签"概念 | 无 | 需要群组+考勤活动定义"谁该来" |
| 教师群组 | 无 | 新增：群组 CRUD + 关联考勤活动 |
| 教师筛选维度 | 仅关键词+状态 | 增加：日期/班级/活动/学号/姓名 |
| 参与/未参与次数 | 无 | 表格列增加"参与次数""缺签次数""到课率" |
| 导出 Excel | 有 | 保留，适配新字段 |

## 理解确认（待用户核实）

### 合照识别流程

```
教师选择某活动
    │
    ▼
上传合照（<1000MB）
    │
    ▼
后端检测人脸 + 识别 + 去重
    │
    ▼
返回：原图 + 每张人脸绘制边框+编号
    │
    ▼
页面展示：
  ├─ 上半部分：标注后的合照（人脸框+编号）
  └─ 下半部分：识别结果列表
       │
       ├─ 已识别学生：编号 / 学号 / 姓名 / 班级 / 置信度 / 情绪
       │     状态：✅ 自动匹配
       │
       └─ 教师手动补充：
            └─ 搜索框（姓名/学号查询）→ 手动添加未识别到的学生
                状态：✏️ 手动补录

    ──→ [保存] 提交本次合照考勤 → 写入数据库
```

### 与当前实现的差异

| | 当前 | 改为 |
|---|---|---|
| 返回形式 | JSON 结果列表 | 标注后的图片（人脸框+编号）+ 列表 |
| 教师补录 | 无 | 搜索姓名/学号，手动添加漏识别学生 |
| 编号映射 | 无 | 图片上的编号与列表一一对应 |
| 活动绑定 | activity_name 文本框手打 | 从已发布的活动列表中选择 |
| 图片大小限制 | 无限制 | < 1000MB |

### 后端改动

| 改动 | 说明 |
|---|---|
| `POST /group-photo/recognize` | 返回增加标注后的图片（base64 或新图片路径），每张人脸带编号 |

| `GET /group-photo/annotated/{id}` | 可选：获取标注后的合照图片 |
