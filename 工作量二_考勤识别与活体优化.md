# 工作量二：考勤识别与活体优化

> 定位：围绕“考勤签到是否准确、安全、好用”的优化工作。  
> 这是验收时最容易被问到的核心功能之一。

---

## 一、工作范围

本工作量主要负责：

1. 教师端考勤签到页面。
2. 摄像头调用、拍照上传、自动捕捉。
3. 人脸检测、人脸特征提取、人脸库比对。
4. 活体检测开关、随机动作挑战。
5. 手动捕捉和自动捕捉的交互优化。
6. 考勤结果记录和签到照片保存。
7. 考勤记录查询、删除、批量删除、导出。

---

## 三、关键代码说明

### 3.1 前端考勤页面

文件：`frontend/src/views/Attendance.vue`

负责：

- 调用摄像头：`navigator.mediaDevices.getUserMedia`
- 通过 `canvas.toBlob` 把当前帧转成图片文件。
- 调用后端 `attendanceApi.checkIn`。
- 展示识别结果：学号、姓名、班级、置信度、活体分、情绪、时间。
- 自动捕捉时持续检测画面，降低人工操作成本。
- 活体动作提示统一显示为短文案，如“请张嘴”“请微笑”。

最近优化点：

| 优化 | 说明 |
|---|---|
| 手动捕捉动作提示 | 摄像头画面上方显示当前活体动作 |
| 自动捕捉提示稳定 | 不再在“请张嘴后完成考勤”和“请张嘴”之间闪动 |
| 自动捕捉请求节流 | 自动检测间隔设置为 0.5 秒，失败时不连续弹窗 |
| 签到后刷新活体动作 | 手动和自动捕捉完成后都会刷新下一组挑战动作 |

验收讲法：

> 前端只负责采集图像和显示状态，真正的人脸识别、活体检测和入库都由后端完成。

### 3.2 后端考勤接口

文件：`backend/app/api/attendance.py`

核心接口：

| 接口 | 作用 |
|---|---|
| `POST /api/attendance/check-in` | 上传当前帧，完成识别、活体检测、写入考勤记录 |
| `POST /api/attendance/verify-action` | 验证当前活体动作是否完成 |
| `GET /api/attendance/liveness-challenge` | 随机返回一组活体动作 |
| `GET /api/attendance/records` | 查询考勤记录 |
| `GET /api/attendance/export` | 导出考勤记录 |
| `DELETE /api/attendance/records/{id}` | 教师删除单条记录 |
| `POST /api/attendance/records/batch-delete` | 教师批量删除记录 |

### 3.3 一次签到的后端流程

```text
读取上传图片
  -> 检测人脸
  -> 检查人脸质量
  -> 提取人脸特征
  -> 活体检测
  -> 动作挑战校验
  -> 情绪分析
  -> 与学生人脸库比对
  -> 生成 AttendanceRecord
  -> 成功时生成 EmotionRecord
  -> 保存签到照片
  -> 返回结果给前端
```

验收讲法：

> 识别不是只看一张脸像不像，还同时看活体分和动作是否完成。只有身份匹配、置信度达标、活体通过，才算成功。

### 3.4 人脸识别

主要文件：

- `backend/app/services/face_service.py`
- `backend/app/api/students.py`
- `backend/app/api/attendance.py`

实现思路：

1. 学生人脸照片录入后，后端提取特征向量，保存到 `students.face_encoding`。
2. 签到时对当前图片提取特征。
3. 与所有已录入学生的人脸特征做相似度比对。
4. 如果最高匹配分超过阈值，就认为识别成功。

答辩口径：

> 学生信息和人脸特征存在同一张 `students` 表中，签到时会加载全部已录入特征作为候选集，然后取最高相似度作为匹配结果。

### 3.5 活体检测和动作挑战

主要文件：

- `backend/app/services/liveness_service.py`
- `backend/app/api/attendance.py`

设计：

| 层次 | 作用 |
|---|---|
| 活体分 | 判断画面是否像真实人脸，而不是照片或屏幕 |
| 随机动作 | 要求用户完成微笑、转头、张嘴等动作 |
| 会话状态 | 后端保存当前用户本轮需要完成的动作和步骤 |

答辩口径：

> 单纯的人脸识别可能被照片欺骗，所以我们加了活体分和随机动作挑战。随机动作每次不固定，可以提高视频重放和静态照片攻击的成本。

## 五、老师可能问的问题

### Q1：识别成功的判断条件是什么？

答：首先要检测到清晰人脸，然后提取特征并和学生人脸库比对，置信度达到阈值；如果开启活体检测，还必须活体检测和动作挑战通过。

### Q2：为什么需要活体检测？

答：只做人脸识别容易被照片或屏幕翻拍欺骗。活体检测和动作挑战用于判断画面中的对象是不是正在配合操作的真人。

### Q3：手动捕捉和自动捕捉有什么区别？

答：手动捕捉由教师点击按钮提交当前帧；自动捕捉会周期性检查画面，有人脸且动作通过后自动提交。两者最后调用的是同一套后端识别和入库逻辑。

### Q4：为什么自动捕捉要做 0.5 秒间隔？

答：如果请求太频繁，页面提示会闪烁，后端压力也会增大。0.5 秒是体验和实时性的折中，既能持续检测，也不会频繁弹错误。

### Q5：失败记录会不会保存？

答：会。失败也会保存到 `AttendanceRecord`，并记录失败原因，方便教师查看和排查。

## 六、当前边界

当前版本把考勤签到作为教师端功能，学生端主要查看自己的记录和维护人脸信息。需求中提到的“识别到本人后学生主动提交考勤”属于后续可扩展方向，需要拆分识别接口和正式提交接口。
