# 角色权限设计

> 更新时间：2026-05-14  
> 依据：`需求分析.md`、当前前后端代码实现、明天验收讲解需要

---

## 一、设计目标

本系统采用“教师端管理 + 学生端自助查看/维护”的双角色权限设计。权限设计的核心目标不是把所有页面简单隐藏，而是同时做到：

1. 前端路由层控制入口，避免学生看到教师管理功能。
2. 后端接口层做最终校验，防止绕过页面直接请求接口。
3. 学生数据按 `student_id` 隔离，只能看到与自己绑定的数据。
4. 教师拥有班级管理、考勤组织、合照识别、统计导出等管理能力。

---

## 二、角色定义

| 角色 | 身份来源 | 账号创建方式 | 默认入口 |
|---|---|---|---|
| 教师 | 系统默认管理账号 | 首次启动自动创建 `teacher / 123456` | 总览面板 |
| 学生 | 与 `students` 表中学生信息绑定 | 教师新增学生或批量导入人脸时自动创建，默认密码 `123456` | 我的人脸 |

代码对应：

| 位置 | 作用 |
|---|---|
| `backend/app/models/user.py` | `User.role` 区分 `teacher` 和 `student`，`student_id` 绑定学生 |
| `backend/app/main.py` | 首次启动创建默认教师账号 |
| `backend/app/api/students.py` | 新增学生、批量导入人脸时自动创建学生账号 |
| `frontend/src/router/index.js` | 根据角色控制页面访问和登录后的默认跳转 |

## 四、前端权限控制

前端使用 Vue Router 和 Pinia 保存登录状态。

| 文件 | 实现点 |
|---|---|
| `frontend/src/stores/auth.js` | 登录后保存 `token`、`username`、`role` |
| `frontend/src/router/index.js` | `meta.role` 限制教师/学生页面访问 |
| `frontend/src/layout/MainLayout.vue` | 菜单根据 `auth.role` 显示不同入口 |
| `frontend/src/views/Login.vue` | 登录后教师跳转总览，学生跳转“我的人脸” |

关键逻辑：

```js
function homePath(role) {
  return role === 'student' ? '/my-profile' : '/dashboard'
}

if (to.meta?.role && auth.role !== to.meta.role) return homePath(auth.role)
```

答辩口径：

> 前端主要负责“体验层”的权限控制，比如隐藏不该看的菜单、限制页面跳转。但前端不能作为安全边界，所以后端还要再校验一次。

## 五、后端权限控制

后端使用 JWT 认证和依赖注入实现接口权限。

| 文件 | 实现点 |
|---|---|
| `backend/app/api/deps.py` | `get_current_user` 解析 token，`require_teacher` 限制教师接口 |
| `backend/app/api/auth.py` | 登录、注册、修改密码 |
| `backend/app/api/students.py` | 学生管理接口大多加 `Depends(require_teacher)` |
| `backend/app/api/attendance.py` | 教师签到、删除记录需要教师权限；查询记录按角色过滤 |
| `backend/app/api/statistics.py` | 情绪统计按角色过滤 |
| `backend/app/api/emotions.py` | 学生只能查询自己的考勤情绪记录 |

核心校验：

```python
def require_teacher(user: User = Depends(get_current_user)) -> User:
    if user.role != "teacher":
        raise HTTPException(status_code=403, detail="仅教师角色可操作")
    return user
```

学生数据隔离示例：

```python
if user.role == "student" and user.student_id:
    query = query.filter(AttendanceRecord.student_id == user.student_id)
```

答辩口径：

> 我们不是只在页面隐藏按钮，而是在后端接口层用 `require_teacher` 做教师权限校验。学生相关查询会额外按 `student_id` 过滤，所以即使学生改请求参数，也只能拿到自己的数据。

## 六、账号与密码设计

| 场景 | 设计 |
|---|---|
| 教师账号 | 系统启动时自动创建 `teacher / 123456` |
| 学生账号 | 教师新增学生或批量导入人脸时自动创建 |
| 学生默认密码 | `123456` |
| 密码存储 | 使用 bcrypt 哈希，不保存明文 |
| 教师重置学生密码 | 学生管理页面提供“重置密码”，重置为 `123456` |
| 学生修改密码 | “我的账户”区域可修改自己的密码 |

代码对应：

| 位置 | 说明 |
|---|---|
| `backend/app/core/security.py` | `hash_password`、`verify_password` |
| `backend/app/api/students.py` | `_ensure_student_user` 自动创建/绑定学生账号 |
| `backend/app/api/auth.py` | `/auth/change-password` 修改当前账号密码 |
| `frontend/src/views/Students.vue` | 教师重置学生密码 |
| `frontend/src/views/MyProfile.vue` | 学生修改自己的密码 |

## 八、老师可能问的问题

### Q1：你们怎么防止学生访问教师页面？

答：前端路由里给教师页面加了 `meta.role = 'teacher'`，学生访问会自动跳到自己的页面。更重要的是，后端教师接口还用了 `require_teacher`，即使学生直接构造请求，也会返回 403。

### Q2：学生怎么只能看到自己的记录？

答：登录 token 解析出当前用户后，如果角色是 `student`，后端查询会加 `student_id == 当前用户绑定学生ID` 的条件。考勤记录、情绪明细、情绪统计都做了这个过滤。

### Q3：学生账号怎么来的？

答：教师新增学生或批量导入人脸时，后端会调用 `_ensure_student_user`，用学号作为用户名，默认密码是 `123456`，同时绑定 `student_id`。

### Q4：密码有没有明文保存？

答：没有。`backend/app/core/security.py` 里使用 bcrypt 对密码做哈希，数据库里保存的是 `password_hash`。

### Q5：为什么学生不能自己进入考勤签到页面？

答：当前验收版本是“教师端组织考勤”，教师打开摄像头进行签到识别。学生端主要负责查看自己的信息、维护人脸照片、查看自己的记录和统计。需求里学生自助签到是可扩展方向，但本版本把核心考勤闭环放在教师端。
